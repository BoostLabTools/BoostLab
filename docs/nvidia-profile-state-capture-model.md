performed in this phase. No NVIDIA profile restore is performed in this phase.
No NVIDIA Profile Inspector execution is approved. No `.nip` import or export
is approved. No production profile operation is approved. No runtime behavior

NVIDIA App Path B order remains:

`Driver Install Latest -> Nvidia Settings -> Hdcp -> P0 State -> Msi Mode`

Path relationship:

* Path A: `Driver Install Debloat & Settings`

Path B is for users who want to keep or use NVIDIA App features such as
recording or related NVIDIA App features. Future UI must preserve guided
separation between Path A and Path B and prevent accidental mixing unless a

## Profile State Concepts

Future NVIDIA profile work must treat these concepts as distinct:

* NVIDIA profile database/state: the driver-controlled profile store and any
  related state that influences NVIDIA Control Panel or driver profile behavior.
* Global driver profile: the base profile that applies broadly before
  application-specific overrides.
* Application-specific profiles: profile entries scoped to executable or
  application identities.
* DRS/Profile Inspector managed settings: setting IDs, names, values, and value
  types exposed through NVIDIA Profile Inspector or an equivalent approved
  model.
* Generated `.nip` profile files: files generated by BoostLab from approved
  source content or templates.
* Imported `.nip` profile files: files consumed by Profile Inspector or a later
  approved profile import mechanism.
* Exported baseline profile files: pre-change captures used as restore
  candidates.
* Pre-change capture: the profile state captured before any profile import or
  write.
* Post-change verification: comparison of detected profile state after an
  attempted operation.
* Restore candidate: a validated prior capture that may be eligible for a
  user-selected Restore action.
* Profile drift: profile state that changed after capture and before restore or
  verification.
* Profile merge conflict: a future condition where an import would overwrite or
  conflict with existing user/application profile state.
* Unsupported profile setting: a setting ID, name, value type, or value not
  supported by the detected driver/Profile Inspector version.
* NVIDIA App compatibility note: Path B is explicitly for users keeping NVIDIA
  App features; profile changes must not silently break that workflow boundary.
* Path A / Path B workflow boundary: Path A and Path B state records must remain
  separate and must not be mixed without explicit future approval.

## Capture Model

Future capture requirements:

* Capture must happen before any profile import or profile write.
* Capture must use an approved and verified external tool or an approved
  API/model.
* Capture output must be written only to bounded BoostLab state paths.
* Capture output must be hashed.
* Capture output must be linked to the exact workflow step and source checksum.
* Capture must include driver version and NVIDIA device identity.
* Capture must fail closed if tool/provenance is not approved.
* Capture must not run from untracked temp paths.
* Capture must not be mixed across Path A and Path B without explicit design
  approval.
* Capture must record whether the profile target is global or
  application-specific.
* Capture must report unsupported settings, unreadable state, or ambiguous
  profile targets as non-approval conditions until a future policy handles them.

## Restore Model

Future restore requirements:

* Restore must only use a validated prior capture.
* Restore must verify matching ToolId, WorkflowId, StepId, SourceChecksum,
  driver identity, GPU identity constraints, and capture integrity.
* Restore must require explicit user confirmation.
* Restore must not be confused with Default. Default means an approved default
  behavior from the source or migration record. Restore means returning to a
  captured previous profile state.
* Restore must integrate with Restore Selection UI / Runtime.
* Restore must verify post-restore profile state.
* Restore must log every attempted, skipped, denied, or completed restore.
* Restore must fail closed on missing capture, hash mismatch, identity mismatch,
  unsupported driver version, ambiguous target, profile drift, or profile merge
  conflict.
* Restore eligibility must be visible in Latest Result before any future Restore
  button is exposed.

## Generated And Imported `.nip` Model

Future generated or imported `.nip` files require:

* Bounded generated file path.
* Generated artifact ownership metadata.
* Source script and step linkage.
* Content hash.
* Profile scope.
* Import target.
* Provenance of input template if any.
* Generation method.
* Cleanup or quarantine policy.
* Validation before import.
* Verification after import.
* Rollback capture before import.
* No approval in this phase.

The generated `.nip` file in `Nvidia Settings` must not be treated as a generic
temp file. It is a generated profile artifact that can mutate NVIDIA profile
state when imported, so it needs its own bounded identity, content hash, profile
scope, and restore plan.

## NVIDIA Profile Inspector Model

Future NVIDIA Profile Inspector usage requires:

* Approved artifact provenance.
* Immutable source or official/trusted source decision.
* Pinned version.
* SHA-256.
* Signer/publisher validation if available.
* File size bounds.
* Bounded install or extraction path.
* Execution descriptor.
* Allowed arguments.
* Expected exit codes.
* No execution from untracked paths.
* Preflight checks.
* Post-operation verification.
* No approval in this phase.

Profile Inspector must not be run from a mutable download, unverified temp path,
or generic executable handoff. Any future execution must flow through the
download provenance, installer/executable execution, process handling, profile
capture, and Action Plan foundations.

## Future UI And Runtime Expectations

Future UI/runtime behavior must include:

* Action Plan must show profile capture and restore intent.
* The user must see whether a profile capture exists.
* The user must see whether Restore is available or denied.
* Path A and Path B profile state must be separated.
* Latest Result must include capture id, status, verification, and restore
  eligibility.
* Activity Log must include profile operation decisions and refusal reasons.
* UI must show `NotImplemented` / foundation-only status until future
  implementation.
* Profile operation failures must remain structured and copyable.
* A missing approved capture model must block future profile writes/imports
  rather than silently proceeding.
